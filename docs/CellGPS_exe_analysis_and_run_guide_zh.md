# CellGPS.exe 构建分析与运行说明

## 1. 结论先说

这份 `Y:\long\projects\Packagedevelopment\CellGPS\sfplot\dist\CellGPS.exe` 基本可以判断为一个 PyInstaller `--onefile` 风格的 Windows GUI 打包产物。

它的“standalone”结论是：

- 可以视为“对普通 Windows 用户来说可独立运行”的单文件程序。
- 更准确地说，它是“无需额外安装 Python/Conda 的单文件启动器”，而不是“完全不解压、完全零依赖”的纯原生单二进制。
- 目标机器需要满足基本条件：`Windows 64-bit`、可写 `%TEMP%` 临时目录、未被杀毒软件/SmartScreen 阻止。

如果用户满足上面这些条件，那么把 `CellGPS.exe` 单独拷到另一台电脑后，原则上是可以运行的。

## 2. 本次检查到的直接证据

### 2.1 产物形态

- `dist` 目录里只有两个文件：
  - `CellGPS.exe`
  - `error.log`
- `CellGPS.exe` 体积约 `425 MB`
- 可执行文件是 `Windows GUI` 子系统
- 架构是 `x64`

这说明它不是依赖旁边一堆 DLL 的 `onedir` 包，而是倾向于单文件包。

### 2.2 内部打包内容

对 exe 的字符串扫描可以直接看到这些关键内容已经被打进包里：

- `python312.dll`
- `vcruntime140.dll`
- `tcl8.6`
- `tk8.6`
- `splash.png`
- `scanpy`
- `spatialdata_io`
- `geopandas`
- `shapely`
- `h5py`

这说明它不是一个只负责“调用本机 Python”的空壳，而是把 Python 运行时和一大批科学计算依赖一起塞进去了。

### 2.3 运行行为

实际启动该 exe 时，会在当前用户的临时目录下创建 `_MEI...` 目录，并且其中能看到 `tcl8.6` / `tk8.6` 目录。

这是 PyInstaller `onefile` 的典型行为，说明程序启动时会先把内部资源解压到 `%TEMP%` 再运行。

### 2.4 当前机器上的复测结果

- 我重新启动过这份 `CellGPS.exe`
- 启动后进程不会立即退出
- 现有 `error.log` 没有被更新

这表示“当前这份 exe 在我这台机器上没有复现立即崩溃”的情况。

## 3. 它能不能算 standalone

### 3.1 可以算 standalone 的部分

从分发角度看，它满足下面几点：

- 不要求用户自己安装 Python
- 不要求用户自己安装 Conda 环境
- 不要求用户手动放置一整套源码目录
- 分发时理论上只需要给用户一个 `CellGPS.exe`

所以如果你的意思是“别人下载一个 exe 到 Windows 电脑上就能点开跑”，答案是：

**大体上可以。**

### 3.2 它不是完全无条件 standalone

它仍然依赖目标机器上的这些环境条件：

- 必须是 `64-bit Windows`
- 必须允许创建和写入 `%TEMP%`
- 必须允许从临时目录加载解压出的 DLL / pyd / Tcl-Tk 资源
- 如果是从浏览器或网盘下载，可能会被 Windows SmartScreen 或企业杀毒软件拦截

所以更准确的表述应该是：

**它是“Windows x64 单文件自带 Python 运行时”的 standalone，不是“任意电脑任意环境都 100% 免问题”的 standalone。**

## 4. 我认为这个 build 目前存在的问题

### 4.1 `dist` 里带着历史 `error.log`

当前 `dist` 目录里有一个 `error.log`，内容记录过一次启动时的 Tk 根窗口初始化错误：

- `RuntimeError: Too early to create variable: no default root window`

这次复测没有复现这个错误，所以它更像是旧日志，不一定代表当前 exe 仍然有这个问题。

但它会带来两个现实问题：

- 用户看到这个日志会误以为“当前版本一启动就坏了”
- 发布包不够干净，容易混淆测试结论

建议正式发版前把这个旧 `error.log` 删除掉，再重新做一次干净分发。

### 4.2 启动 hook 比较脆

仓库里有自定义 PyInstaller 启动 hook，专门处理：

- Tcl/Tk 路径
- Splash 图
- 全局异常日志

这说明 GUI 启动链路经过了额外定制。它能工作，但也意味着：

- 启动过程比普通 Tk 程序更脆弱
- 若换 Python 版本、换打包参数、换 PyInstaller 版本，可能再次引入启动顺序问题

### 4.3 GUI 导入链过重

GUI 入口文件只用到了少数功能，但它通过 `from sfplot import ...` 导入顶层包，而顶层包又会继续导入很多模块。

这会带来两个副作用：

- 打包体积很大
- 启动时把大量并非 GUI 必需的依赖也一起拉起来，增加跨机器兼容风险

换句话说：

这个 build 更像“把整个科研 Python 环境塞进了一个 exe”，而不是“为 GUI 精简过的发布版”。

### 4.4 当前 GUI 没有导出结果按钮

从代码看，GUI 目前的行为是：

- 计算结果后把热图显示在界面里
- 不自动保存 PDF
- 也没有明显的“Save/Export”按钮

这不是 standalone 问题，但它是非常实际的交付问题：

**用户虽然能运行，但不一定能方便地把结果保存出来。**

### 4.5 构建流程不可复现

仓库里能看到 PyInstaller 的 hook 文件，但没有看到：

- 固化的 `.spec` 文件
- 明确的打包命令脚本
- 清晰的发布流程记录

所以当前这份 exe 可以运行，不代表别人以后还能稳定地重新打出完全一样的版本。

如果要正式对外发布，建议把构建流程固定下来。

## 5. 分发给别人时，我建议的口径

建议不要说：

- “这是绝对纯净、完全不依赖任何系统条件的 exe”

建议这样说：

- “这是面向 Windows 64 位的单文件版 CellGPS，已内置 Python 和主要依赖，通常无需额外安装环境即可运行。首次启动会稍慢，因为程序会先在临时目录解压运行时文件。”

这个表述更准确，也更符合它现在的打包方式。

## 6. 详细运行说明

### 6.1 适用系统

- Windows 10 或 Windows 11
- 64 位系统
- 建议至少 8 GB 内存
- 需要有本地磁盘临时目录写权限

### 6.2 获取程序

建议把 `CellGPS.exe` 复制到本地磁盘再运行，不要直接在网络盘里双击。

推荐位置示例：

- `C:\Users\<用户名>\Desktop\CellGPS\CellGPS.exe`
- `C:\Users\<用户名>\Downloads\CellGPS\CellGPS.exe`

不建议：

- 直接从 `Y:` 网络盘运行
- 直接从压缩包内部双击运行

### 6.3 首次运行前建议

如果文件是从邮件、网盘、浏览器下载得到的，先做下面检查：

1. 右键 `CellGPS.exe`
2. 打开“属性”
3. 如果底部有“解除锁定”或 `Unblock` 选项，勾上后点击“确定”

如果电脑装有公司安全软件，第一次运行可能被拦截。遇到这种情况，通常需要：

- 允许该程序运行
- 或把程序加入白名单

### 6.4 启动程序

双击 `CellGPS.exe` 即可。

注意：

- 第一次启动可能比普通桌面软件慢
- 原因是它会先把内部运行时解压到 `%TEMP%`
- 启动期间可能先出现短暂加载过程

如果双击后没有立刻弹出主界面，请至少等待 `10-30` 秒再判断是否失败。

### 6.5 程序界面结构

从代码看，主界面有两个 tab：

- `CSV Heatmap`
- `Xenium Heatmap`

## 7. `CSV Heatmap` 使用方法

### 7.1 输入文件要求

该模式读取一个普通 CSV 文件。

CSV 至少需要包含这些列名：

- `x`
- `y`
- `celltype`

如果列名不一致，当前 GUI 没有提供列名映射设置，程序会直接报错。

### 7.2 操作步骤

1. 打开程序
2. 进入 `CSV Heatmap` tab
3. 点击 `Select CSV File`
4. 选择你的输入 CSV
5. 选择后，点击 `Plot CSV Heatmap`
6. 等待程序读取数据、计算距离并绘图
7. 结果会显示在右侧显示区域
8. 可通过 `Zoom` 调整缩放

### 7.3 注意事项

- 该模式当前是“显示结果”为主
- 代码中没有看到明确的导出按钮
- 也没有默认自动保存 PDF

所以如果你希望用户能直接导出结果，当前 GUI 还不够完整，需要后续补保存功能。

## 8. `Xenium Heatmap` 使用方法

### 8.1 Xenium 目录要求

选择的 Xenium 数据目录应至少满足下面之一：

- 已经存在 `analysis/clustering/.../clusters.csv` 和 `analysis/umap/.../projection.csv`
- 或目录下有 `analysis.tar.gz`
- 或目录下有 `analysis.h5`

此外，程序还会读取 Xenium 主数据，并依赖 `cell_id`、`Cluster`、`spatial` 等信息正确建立。

### 8.2 Selection CSV 要求

第二个 CSV 文件必须包含一列，列名严格为：

- `Cell ID`

程序会用这列去筛选 Xenium 数据中的细胞。

### 8.3 操作步骤

1. 打开程序
2. 进入 `Xenium Heatmap` tab
3. 点击 `Select Xenium Dir`
4. 选择 Xenium 数据目录
5. 点击 `Load Xenium Data`
6. 等待程序完成 Xenium 数据载入
7. 点击 `Select Selection CSV`
8. 选择包含 `Cell ID` 列的 CSV
9. 点击 `Plot Xenium Heatmap`
10. 等待结果显示在右侧区域

### 8.4 特别注意

如果所选 Xenium 目录中只有 `analysis.tar.gz` 而没有已解压的 `analysis` 文件夹，程序会尝试把压缩包直接解压到你选中的数据目录。

这意味着：

- 程序会修改原始数据目录内容
- 首次读取可能更慢
- 目录需要可写权限

如果你不希望程序改动原始数据，建议先手动复制一份测试数据到本地，再让程序读取。

## 9. 常见问题排查

### 9.1 双击没反应

优先检查：

- 是否被 SmartScreen 拦截
- 是否被杀毒软件拦截
- 是否还在首次解压阶段
- 是否在网络盘直接运行

建议做法：

- 先复制到本地磁盘
- 等待至少 30 秒
- 再看是否弹窗

### 9.2 能启动但立刻报错

先看 exe 同目录下是否生成或更新了：

- `error.log`

这是当前版本的主要错误日志文件。

### 9.3 在别的电脑能不能运行

一般可以，但前提是：

- 对方电脑是 Windows x64
- 本地 `%TEMP%` 可写
- 安全软件没有拦截
- 输入数据格式符合要求

### 9.4 为什么第一次启动慢

因为这是 `onefile` 打包，程序每次启动前都需要在临时目录解包一批运行时文件。

### 9.5 为什么 exe 这么大

因为它把大量 Python 运行时和科学计算依赖都打包进去了，包括：

- `scanpy`
- `spatialdata_io`
- `geopandas`
- `shapely`
- `h5py`

这会换来“无需用户安装环境”的便利，但代价就是：

- 包体大
- 启动慢
- 更容易被安全软件重点检查

## 10. 对正式发布的建议

如果你准备把这份程序发给别人长期使用，建议在正式发布前做这些整理：

1. 删除旧的 `error.log`
2. 补一份固定的 PyInstaller `.spec` 或打包脚本
3. 给 GUI 增加“保存结果”按钮
4. 尽量避免 GUI 启动时导入不必要的大模块
5. 用一台没有 Python/Conda 的干净 Windows x64 机器做最终冒烟测试
6. 发布时附带这份运行说明

## 11. 最终判断

最终判断可以概括成一句话：

**这份 `CellGPS.exe` 已经具备“单文件分发到另一台 Windows 64 位电脑上运行”的基本条件，能算 standalone；但它仍然是 PyInstaller onefile 风格的自解压程序，不是完全零环境条件的原生绿色软件。**

如果只是内部测试或小范围分发，已经够用。

如果要正式对外发布，建议先把日志、导出功能、构建可复现性这三件事补齐。
